# RTSP-Pusher V2 开发计划

## 1. 目标

V2 的目标不是重写当前项目，而是在 V1 已经能完成音视频采集、编码、RTSP 推流和基础重连的基础上，把“可运行”推进到“可验证、可观测、可长期稳定运行”。

本计划基于以下现状判断：

- 视频采集和视频编码主链路已经基本成立，稳定段约 30fps，无视频丢帧。
- 音频采集和 AAC 编码已经接入，稳定段音频编码持续增长。
- mux 已能同时写入音视频 packet，稳定段未出现写入错误。
- 低延时参数已部分启用，包括 `ultrafast`、`zerolatency`、`max_b_frames=0`、小队列和 `muxdelay=0`。
- 当前仍存在 V2 必须处理的问题：码率统计恒为 0、日志出现多次 VBV underflow、音视频缺少统一同步基准和漂移观测、低延时没有端到端指标、重连和长稳验证还不充分。

## 2. V2 范围

### 2.1 纳入范围

- 音视频同步基准和同步误差统计。
- 视频码控和低延时编码参数稳定化。
- 音频采集、ring buffer、静音补偿和溢出处理。
- RTSP mux 写入、重连、队列清理和旧包隔离验证。
- 运行统计、CSV、日志和长稳测试报告。
- 可复现的验收脚本和人工测试流程。

### 2.2 暂不纳入范围

- 不切换到 Qt 或新增 GUI。
- 不引入复杂插件系统。
- 不重写 FFmpeg/SDL 主链路。
- 不同时支持多路推流。
- 不在 V2 强制实现硬件编码，除非软件编码无法满足验收指标。

## 3. 总体策略

V2 按“先测量，再修正，再固化”的顺序推进：

1. 先补齐统计指标，确保问题可定位。
2. 再处理音视频同步和码控两类核心质量问题。
3. 最后做长稳、断线恢复和发布验收。

每个阶段必须有日志、CSV 或播放端证据，不以“代码看起来合理”作为完成标准。

## 4. 阶段计划

| 阶段 | 名称 | 核心目标 | 退出标准 |
|------|------|----------|----------|
| V2.0 | 基线审计 | 固化当前运行表现和问题清单 | 有一份可复现基线报告 |
| V2.1 | 可观测性补齐 | 修复码率统计，增加关键延时和队列指标 | CSV 能反映真实码率、队列、耗时 |
| V2.2 | 音视频同步 | 建立统一启动基准和同步误差观测 | 长稳测试无明显 A/V 漂移 |
| V2.3 | 低延时和码控 | 消除或显著降低 VBV underflow，控制端到端延迟 | 码控稳定，延迟不随时间累积 |
| V2.4 | 音频健壮性 | 处理 ring overflow/underrun 和设备异常 | 音频断续可定位，不破坏主链路 |
| V2.5 | 恢复和长稳 | 强化断流、重连、退出和资源释放 | 多次断流恢复，30 分钟以上稳定 |
| V2.6 | 发布验收 | 整理文档、脚本和验收报告 | V2 验收清单全部通过 |

## 5. V2.0 基线审计

### 目标

把当前版本的实际表现记录为 V2 起点，避免后续优化没有对照。

### 实现项

- 保留当前可运行二进制、启动命令、日志和 CSV。
- 记录测试环境：系统、CPU、屏幕分辨率、RTSP server、播放端。
- 记录至少 10 分钟推流数据。
- 统计以下指标：
  - 视频采集 fps、编码 fps、丢帧数。
  - 音频采集字节数、AAC 编码帧数。
  - RTSP 写包数、写错误数、重连次数。
  - rawQueue、encQueue 最大值。
  - `encodeMaxUs`、`muxWriteMaxUs` 最大值。
  - VBV underflow 次数。

### 验收标准

- 生成 `docs/v2-baseline-report-YYYY-MM-DD.md`。
- 报告中明确列出当前可接受项和不可接受项。
- 不修改业务代码。

## 6. V2.1 可观测性补齐

### 目标

让运行质量可以直接从日志和 CSV 判断。

### 实现项

- 修复 `PusherStats::bitrateKbps`，按最近统计窗口计算实际输出码率。
- 区分视频包和音频包计数。
- 增加当前窗口 fps，而不仅是累计平均 fps。
- 增加以下时间点：
  - 程序启动到首帧采集。
  - 首帧采集到首个视频包编码完成。
  - 首个编码包到首个 RTSP 写包成功。
  - 音频采集开始到首个 AAC 包。
- 增加队列水位统计：
  - rawQueue 当前值和最大值。
  - encQueue 当前值和最大值。
  - audio ring 当前字节数和最大值。
- 日志中明确标记统计窗口，例如 `statsWindowSec=5`。

### 验收标准

- CSV 中 `bitrateKbps` 不再恒为 0。
- 推流 10 分钟后，CSV 可直接判断码率、队列、错误和耗时趋势。
- 日志和 CSV 字段含义在文档中同步更新。

## 7. V2.2 音视频同步

### 目标

让音视频同步从“各自 PTS 单调”升级为“同一启动基准下可观测、可判断”。

### 实现项

- 新增或补全 `AVSyncClock`。
- 在 pipeline 启动时建立统一 `startUs`。
- 视频帧记录 `captureTimeUs`，编码 PTS 仍可保持 CFR，但需要保留 wall-clock 映射。
- 音频 ring buffer 记录首个采样块对应的时间。
- 统计 A/V offset：
  - 最近视频 PTS 对应时间。
  - 最近音频 PTS 对应时间。
  - `avOffsetMs = audioClockMs - videoClockMs`。
- 对 reconnect 后的首包 PTS 和 serial 做清晰重置。
- 记录 PTS/DTS 单调性异常。

### 验收标准

- 10 分钟运行中无 non-monotonic DTS/PTS。
- 日志或 CSV 可看到 `avOffsetMs`。
- 长稳运行中 `avOffsetMs` 不持续单向扩大。
- 播放端主观检查无明显音画不同步。

## 8. V2.3 低延时和码控

### 目标

减少当前日志中的 VBV underflow，并证明延迟不会随运行时间累积。

### 实现项

- 梳理 x264 当前参数，明确 CRF、bitrate、maxrate、bufsize 的优先级。
- 评估是否改为更稳定的低延时 ABR/CBR 配置。
- 补齐关键参数：
  - `keyint=30`
  - `min-keyint=30`
  - `scenecut=0`
  - `nal-hrd` 是否适合当前 RTSP server
  - `threads` 策略是否固定
- 对 1920x1080 30fps 分别测试 4Mbps、6Mbps、8Mbps。
- 增加 VBV underflow 计数统计。
- 记录播放端观察延迟：
  - 开播延迟。
  - 鼠标移动到播放端可见延迟。
  - 运行 10 分钟后的延迟。

### 验收标准

- VBV underflow 在目标配置下消失，或降低到可接受并被记录。
- rawQueue 和 encQueue 不持续增长。
- 10 分钟后播放延迟不明显高于开播 1 分钟时。
- 最终推荐一组默认编码参数，并说明取舍。

## 9. V2.4 音频健壮性

### 目标

保证音频链路异常时行为明确，不拖垮视频和 RTSP 主链路。

### 实现项

- 统计 audio ring overflow 和 underrun。
- 当 ring buffer 为空时明确策略：
  - 短时间补静音。
  - 长时间无数据记录 warning。
  - 超过阈值时按配置决定继续推无声流或触发恢复。
- 当 ring buffer 满时明确策略：
  - 记录丢弃字节数。
  - 不在 SDL callback 中执行复杂逻辑。
- 支持 `--require-audio` 和 `--no-audio` 的行为文档化。
- 校验不同音频设备：
  - 麦克风输入。
  - 立体声混音。
  - 不存在或禁用设备。

### 验收标准

- 音频设备不可用时，日志明确说明原因。
- 音频短暂中断不导致视频推流崩溃。
- CSV 可看到 audio overflow/underrun 计数。
- 播放端能确认音频存在且无明显断续。

## 10. V2.5 恢复和长稳

### 目标

证明程序在实际网络和服务端异常下能恢复，退出时资源释放完整。

### 实现项

- 重测 Broken pipe 场景。
- RTSP server 停止、恢复各 3 次。
- 推流中断网或阻塞目标端口。
- 退出时检查所有线程 stop/join 顺序。
- reconnect 后检查：
  - serial 是否递增。
  - rawQueue 和 encQueue 是否清空。
  - 旧包是否被丢弃。
  - stats 是否按预期 reset 或累计。
- 明确 `reconnectCount` 是否跨 pipeline 累计。

### 验收标准

- 连续 3 次 RTSP 服务中断后都能恢复推流。
- 每次恢复后播放端重新可见画面和声音。
- 退出无崩溃、无线程悬挂。
- 长稳 30 分钟无队列持续增长、无未解释错误。

## 11. V2.6 发布验收

### 目标

形成 V2 可交付状态。

### 实现项

- 更新 `docs/rtsp-pusher-design.md` 中与实际实现不一致的内容。
- 更新 `docs/rtsp-pusher-development-plan.md` 或保留为 V1 计划并注明历史版本。
- 新增 V2 验收报告。
- 固化推荐启动命令。
- 固化默认配置。
- 整理已知限制。

### 验收标准

- Release 构建通过。
- 使用推荐命令可稳定推流。
- 播放端可看到视频并听到音频。
- CSV 和日志能解释运行状态。
- V2 验收报告中没有未关闭的 P0/P1 问题。

## 12. 优先级清单

| 优先级 | 工作项 | 原因 |
|--------|--------|------|
| P0 | 修复码率统计恒为 0 | 没有码率就无法判断码控和低延时 |
| P0 | 增加 A/V offset 统计 | 当前同步只靠主观播放观察，不够可追溯 |
| P0 | 处理 VBV underflow | 当前日志已暴露编码码控风险 |
| P1 | 音频 overflow/underrun 统计 | 音频异常必须可定位 |
| P1 | 重连后队列和 serial 验证 | 防止旧包污染新连接 |
| P1 | 端到端延迟测试流程 | 低延时必须有可重复验证方法 |
| P2 | WASAPI loopback 预研 | 仅在 SDL 设备不能稳定采集系统声时推进 |
| P2 | 硬件编码预研 | 仅在 CPU 或延迟指标无法达标时推进 |

## 14. 建议执行顺序

1. 先做 V2.0 和 V2.1，不先改同步和码控。
2. 有统计后再做 V2.2 音视频同步。
3. 同步指标稳定后处理 V2.3 码控和低延时。
4. 再补 V2.4 音频健壮性。
5. 最后做 V2.5、V2.6 的长稳和发布验收。

这个顺序的核心原因是：当前项目已经能跑，V2 的风险不在“能不能启动”，而在“出问题时能不能看见、定位并证明已经修好”。
